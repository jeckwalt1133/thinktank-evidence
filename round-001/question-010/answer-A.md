## 阶段 1/3: 初始分析（Alpha）

### 场景重述与约束清单

**已知约束**：
- C1: CEO钦点，2周必须上线
- C2: 调配全公司1/3资源（数量和类型不明）
- C3: 团队新人居多（经验不足）
- C4: 项目期间恰逢国庆长假（7天人力空窗）
- C5: CCTV及各大门户已投广告（不可延期，不可失败）

**隐含约束（需确认）**：
- 广告投放日期是否=上线日期？（如果是"上线前预热广告"，则上线日可能晚于首支广告）
- "1/3资源"包含哪些类型？（开发、测试、运维、设计、市场？）
- 项目的核心功能范围是否可裁减？（"必须上线" vs "必须全部功能上线"）

### 核心主张

**本项目的最大风险不是"时间紧"或"资源少"——而是"不可降级"。CCTV广告已投意味着任何形式的"降级上线"（功能裁剪、体验粗糙、Bug较多）都是不可接受的。这决定了策略选择：不能走"最小化产品范围保上线质量"的标准敏捷路径——必须走"以质量红线反推范围、以冗余对冲不确定性的军事化项目管控路径"。**

### 1. 资源调配方案

#### 1.1 资源需求估算（基于约束反推）

假设项目为典型的中型Web/App功能上线，两周工期正常需要：
- 前端 3人 × 10天 = 30人天
- 后端 3人 × 10天 = 30人天
- 测试 2人 × 10天 = 20人天
- 运维 1人 × 10天 = 10人天
- 产品/设计 2人 × 10天 = 20人天
- 合计约110人天

考虑到新人居多（效率系数 0.5-0.7），实际需投入约160-200人天。两周10个工作日（扣除国庆7天），需配置16-20人。全公司1/3资源若能调动到此量级，理论可行。

#### 1.2 人力资源配置矩阵

| 角色 | 人数 | 配置策略 | 新人比 |
|------|:---:|---------|:------:|
| 核心架构/技术负责人 | 2 | 必须资深 | 0% |
| 前端开发 | 4 | 1资深+3新人，结对编程 | 75% |
| 后端开发 | 4 | 1资深+3新人，结对编程 | 75% |
| 测试 | 3 | 1资深+2新人 | 67% |
| 运维/发布 | 1 | 必须资深 | 0% |
| 产品/设计 | 1-2 | 资深 | 0% |

关键原则：**关键路径上的关键角色必须资深**（架构、运维、产品），用新人填充可并行化且出错了可回滚的模块。

#### 1.3 时间排期（倒推法）

```
D-14 ~ D-10: 需求冻结 + 架构设计 + 环境搭建（资深人员主导）
D-9  ~ D-6:  核心功能开发冲刺（全员）
D-5  ~ D-3:  国庆长假（仅保留on-call值班修复P0 Bug）
D-2  ~ D-1:  集成测试 + 性能压测 + 安全扫描
D-Day:       上线 + 全时监控 + 回滚预案就位
```

关键决策：国庆期间不能完全停工——必须安排on-call值班。如果国庆完全停摆，则只有7个工作日，项目大概率失败。

### 2. 风险矩阵与应对

| 风险ID | 风险描述 | 概率 | 影响 | 等级 | 应对策略 |
|--------|---------|:--:|:---:|:---:|---------|
| R1 | 新人产出质量不达标 | 高 | 高 | 🔴 | 结对编程+每日代码审查+自动化测试门槛 |
| R2 | 国庆期间无人在岗，进度断裂 | 高 | 极高 | 🔴 | 提前安排国庆值班表+激励补偿 |
| R3 | 需求变更（CEO或其他利益方） | 中 | 极高 | 🔴 | D-14后需求冻结，变更需CEO本人审批 |
| R4 | 第三方依赖不可用（API/服务） | 中 | 高 | 🟡 | 提前确认依赖项可用性+准备Mock降级 |
| R5 | 上线时CCTV广告流量远超预期 | 中 | 高 | 🟡 | 提前压测至预估流量3倍+自动扩容配置 |
| R6 | 新人离职（压力过大） | 低 | 高 | 🟡 | 控制加班文化+明确预期+心理安全感 |
| R7 | 发布回滚失败 | 低 | 极高 | 🟡 | 灰度发布+自动回滚脚本+发布前演练 |

### 3. 与营销投放的时间同步策略

#### 3.1 关键假设确认

必须先与市场部确认：CCTV及门户广告的投放排期精确到日。两种常见场景：

- **场景A（广告=上线信号）**：广告播出日=上线日——风险极高，上线必须完美
- **场景B（广告=预热）**：广告提前3-5天播出，上线日在广告周期内——有时间缓冲

#### 3.2 同步策略

- D-5: 向市场部确认广告排期和核心卖点（确保产品功能与广告承诺一致）
- D-3: 完成冒烟测试，向市场部提供"可播广告"确认
- D-1: 灰度环境模拟上线，关键链路验证
- D-Day: 配合广告播出时间精准上线（建议在广告首播前4-6小时完成上线+监控稳定）

#### 3.3 降级方案

如果D-3无法确认上线，必须立即通知市场部启动Plan B——调整广告内容（如从"立即体验"改为"即将上线"）或推迟广告播出。**这条是不可妥协的红线——产品不能拖累已经投放的广告。**

## 阶段 2/3: 双角度批判

### 批判角度一（聚焦组织与人力）

**对"不可降级"假设的批判。** 初始分析的核心前提——"CCTV广告已投意味着不可降级"——需要被严格检查。广告承诺了什么？如果广告只说了品牌名+核心功能概念（而非具体功能列表），那么"降级上线"是可行的——只要核心品牌体验不崩。建议先确认广告的具体内容承诺，而非基于"CCTV=不能失败"的恐惧做决策。过度保守的"质量红线"可能在两周内是不可实现的——届时真正的灾难是"什么都没上线"。

**对新人管理方案的批判。** 结对编程+每日代码审查的时间成本被低估了。结对编程在短期内会降低速度（约30-40%），对2周项目可能是致命的。建议替代方案：新人处理**隔离度高、出错影响可控**的任务（如独立功能模块的UI层、独立的单元测试编写），而非嵌入核心链路。

**对国庆值班的批判。** 国庆值班在劳动法和团队士气层面都是高风险决策。如果团队中新人比例高，强制值班可能导致R6（新人离职）概率大幅上升。建议考虑：国庆前完成所有核心功能开发，国庆期间仅做自动化测试+性能调优（可由资深on-call执行），降低对全员的依赖。

### 批判角度二（聚焦技术与流程）

**对排期的批判。** 初始排期假设D-Day是第14天。但一周7天约束下，国庆（7天）+前后周末，实际可用工作日可能只有5-7天而非10天。必须用精确日历日期反推而非"10个工作日"估算——这是一个致命的排期计算错误。

**对"质量红线"策略的批判。** 两周工期+新人的约束下，"质量红线"可能意味着"不可实现"。更务实的策略是**分级质量目标**：
- 金牌目标：核心用户体验路径零Bug
- 银牌目标：主流程可用，边缘场景有已知Bug但不阻塞
- 铜牌目标：可上线的最低标准

初始分析的"不可降级"导致必须追求金牌——但如果资源不支持金牌，需要CEO亲自做"金牌or延期"的选择。这是项目负责人的核心责任——不只是执行，而是揭示不可行性。

**对技术债的批判。** 两周突击必然积累技术债。初始分析未提及上线后的还债计划——CCTV广告带来的流量高峰过后，系统是否需要紧急重构？建议在计划中明确上线后第1-2周的"技术债偿还Sprint"。

## 阶段 3/3: 整合分析（Alpha 修订）

### 修订核心主张

**本项目的核心矛盾不是"时间vs质量"——而是"广告承诺vs可实现范围"。解决方案不是用力执行初始计划——而是首先搞清楚广告承诺了什么，然后将产品范围精确对齐到广告承诺的最低充分集。任何超出广告承诺的功能都是此刻不可承受的奢侈。**

### 1. 修订后的资源调配方案

#### 第一步：Day 0行动（立即执行，先于任何开发）

1. 获取CCTV及门户广告的精确素材/文案——确认广告承诺的具体功能点
2. 将广告承诺映射为**必须上线的功能清单**（硬范围）
3. 将其他功能标记为"可降级/可推迟"
4. 用精确日历日期（而非工作日估算）反推排期

#### 第二步：精确日历排期

假设5月8日（今日）为D-Day，国庆为10月1-7日：
```
5月8日(D-14): 广告承诺确认+需求冻结（仅1天！）
5月9-11日:    架构+核心开发启动
...需要确认实际日期。本案例国庆为假设参数，实际排期需用具体日期。
```

前提：立即确认项目的实际起止日期（特别是有无预开发期）。

#### 第三步：人力配置（修订）

| 角色 | 人数 | 关键调整 |
|------|:---:|---------|
| 技术负责人 | 2 | 不变 |
| 前端 | 3 | 取消结对编程。新人负责独立UI模块，资深负责核心交互链路 |
| 后端 | 3 | 取消结对编程。新人负责独立数据处理任务，资深负责API网关和核心业务逻辑 |
| 测试 | 3 | 1资深编写自动化测试框架+2新人执行手工测试 |
| DevOps | 1 | 必须资深。负责CI/CD+监控+回滚 |
| 产品 | 1 | 负责需求冻结+每日决策 |

关键调整：**取消结对编程，改为任务隔离**——新人的产出用自动化测试门禁来保证质量，而非实时人工审查。这压缩了同步开销。

### 2. 修订风险矩阵

| 风险ID | 修订 | 新增应对 |
|--------|------|---------|
| R1（新人质量） | 取消结对编程改为测试门禁 | 新人代码合并需通过自动化测试套件+1位资深approval |
| R2（国庆断裂） | 明确国庆前封板 | 国庆前最后工作日完成代码冻结，禁止新功能，仅修P0 Bug |
| R3（需求变更） | 增强 | 需求冻结=广告承诺功能清单。任何新增=CEO本人签发+延期确认 |
| R8（新增） | — | **广告承诺与产品范围不对齐**：概率高/影响极高。应对=Day 0确认+不对齐则立即升级CEO |
| R9（新增） | — | **上线后技术债爆炸**：概率高/影响中。应对=上线后Week1安排技术债偿还Sprint |

### 3. 修订时间同步策略

#### 3.1 四重时间节点对齐

```
广告素材确认 ←→ 产品功能清单冻结（Day 0, 最高优先级）
广告排期确认  ←→ 上线日期锁定（Day 0）
广告首播时间  ←→ 上线窗口（广告首播前6h完成上线+1h监控稳定）
广告高峰期    ←→ 扩容+监控（提前压测至3倍流量）
```

#### 3.2 CEO沟通模板（建议Day 0立即触发）

如果广告承诺的功能超过2周可实现范围：必须立即向CEO呈现"范围-时间-质量"三角困境——三个选项：
- **选项A**: 减少广告承诺的功能→需修改广告（可能不可行）
- **选项B**: 延长上线时间→广告延期（可能不可行）
- **选项C**: 接受降级上线→核心体验在线，边缘功能后续补齐（推荐默认选项）

### 4. 上线后72小时行动计划

| 时间段 | 动作 |
|--------|------|
| 上线后0-2h | 全员监控，回滚脚本就位，与市场部保持实时沟通 |
| 上线后2-24h | 轮班监控，每小时SLA报告，自动扩容规则生效 |
| 上线后24-72h | 降级为on-call，Bug分级修复（P0=1h内，P1=4h内，P2=下周） |
| 上线后Week1 | 技术债评估+偿还计划+事后复盘（Post-Mortem） |

### 5. 修订关键假设

| 假设 | 验证方式 |
|------|---------|
| 广告承诺的功能可在2周内实现核心版本 | Day 0广告素材审查（最高优先级行动） |
| 公司1/3资源=至少12名可全职投入的人员 | Day 0人力资源确认 |
| 第三方依赖在项目期间稳定可用 | Day 0依赖方确认邮件 |
| 新人任务隔离策略有效——自动化测试门禁可保证质量 | D-3代码审查+测试覆盖率检查 |
